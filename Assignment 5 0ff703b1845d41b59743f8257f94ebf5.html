What is a major drawback of recursive lookups when resolving a key in a DHT-based system?

The major drawback of recursive lookups is that the client will not be able to discover errors in communication because if something went wrong, no answer is returned.

Considering that a two-tiered home-based approach is a specialization of a hierarchical location service, where is the root?

The root is collectively formed by all home locations, but it is divided in a manner that ensures each mobile entity has its own root server such that the ratio between the root servers and mobile entities is 1.

The root node in hierarchical location services may become a potential bottleneck. How can this problem be effectively circumvented?

Random bit strings allow partitioning of identifier space, so we can spread partitioned roots for network-wide access.

Give an example of how the closure mechanism for a URL could work.

A URL consists of a protocol, a hostname (i.e. subdomain and top-level domain), a port, a path, a query, and a fragment. The closure mechanism is as follows: we extract each component and look up left to right, we search for the local implementation of the protocol, and we pass the hostname to the local DNS, and for the rest, we apply the hostname resolution.

Explain the difference between a hard link and a soft link in UNIX systems. Are there things that can be done with a hard link that cannot be done with a soft link or vice versa?

Essentially, a hard link points to the same file, and a symbolic link contains a name. The latter allows you to link different disk partitions or even different machines.

High-level name servers in DNS, that is, name servers implementing nodes in the DNS namespace that are close to the root, generally do not support recursive name resolution. Can we expect much performance improvement if they did?

Since high-level name servers in DNS are a global layer, we can expect changes don’t occur often. So, we cannot expect much performance improvement.
