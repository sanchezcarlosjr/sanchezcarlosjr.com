  1. In many layered protocols, each layer has its own header. Surely it would be more efficient to have a single header at the front of each message with all the control in it than all these separate headers. Why is this not done?

Because something can go wrong when we merge headers, for example, some processes might write in the wrong direction, it is harder to detect errors and higher layers must care about how works lower layers. So, it destroys transparency.

  1. Why are transport-level communication services often inappropriate for building distributed applications?

Since it doesn't offer a good API to build distribution transparency, developers are required to pay attention to lower protocols and that is undesirable.

  1. A reliable multicast service allows a sender to reliably pass messages to a collection of receivers. Does such a service belong to a middleware layer, or should it be part of a lower-level layer?

You could implement a multicast service in transport layers or lower-level layers, but it doesn’t offer custom services that would be necessary for software requirements or its scalability, so a middleware layer is a better idea.

  1. One way to handle parameter conversion in RPC systems is to have each machine send parameters in its native representation, with the other one doing the translation, if need be. The native system could be indicated by a code in the first byte. However, since locating the first byte in the first word is precisely the problem, can this work?

We can reduce this problem to agree by using a fixed format for the first byte. A computer sends a byte 0 with the code padding and the receiver reads the location of the code. For example, a big-endian sends 1000000 otherwise 0000000. The receiver reads the first bit and that work independently of the endianness. This is called Byte Order Mark (BOM).

However, communication protocols define and enforce their endianness. So, heterogeneous nodes have a guarantee the endianness or other representations. For example, the Internet Protocol (TCP/IP) suite is defined to be big-endian. Another way is having a broker or proxy that knows the end-node representations and does the conversion correctly.
